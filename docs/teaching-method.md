# 先導課程教學方法

這份文件記錄 Day 1 Prep Course 的設計原則，讀者是講師、助教與未來維護者。

它不是講稿，也不是學員操作手冊。

## 課程定位

先導課程要建立共同工作方式，不提前開始 Day 2 專案。

學生在第一天只需要走過一次縮小版工程流程：

```text
自然語言
→ 結構化資料
→ Agent 修改 repo
→ 人類檢查 diff
→ 前端展示
```

這個流程足夠小，才能讓帳號、工具、協作習慣與安全感一起被測通。

## 設計判準

### 1. 任務必須低風險

第一天的任務不應暴露程度差異，也不該要求學生立即寫功能。

profile 適合當第一個任務，因為它：

- 好理解
- 可公開
- 容易修正
- 能快速看到成果
- 不需要學生自評能力

### 2. Agentic Coding 從上下文開始

這堂課不把 Agent 當成聊天工具，而是讓學生看到：

```text
清楚文字
→ 明確規格
→ repo 修改
→ 可驗證輸出
```

因此自然語言 note、schema、fixed options、validate script 都是教學設計的一部分。

### 3. 版本控制先教 diff

第一天不急著講完 branch、merge、rebase。

更重要的是建立習慣：

```text
Agent 改檔
→ 人看 diff
→ 確認內容
→ 再 commit / PR
```

學生先理解「修改要被審查」，後面才有理由學更多 Git 操作。

### 4. 可視化成果要有成就感，但不能喧賓奪主

Showcase 頁面要讓學生看到自己的資料真的進入 repo 並被渲染出來。

視覺與互動可以豐富，但不能讓專案變成前端框架示範，也不能讓兩小時課程變成除錯地獄。

## 為什麼要有 faction

OS、瀏覽器、編輯器陣營有三個價值：

- 好填，不要求能力排序。
- 有社群感，可以降低課堂緊張。
- 可統計，能自然帶出資料清理和固定代號的重要性。

faction 只能是趣味分類，不應被用來比較強弱。

## 為什麼使用固定代號

自由輸入會讓統計失準：

```text
VS Code
vscode
VSCode
Visual Studio Code
vs code
```

所以資料存固定代號：

```json
"editor": "vscode"
```

畫面再轉成顯示名稱：

```text
VS Code
```

這是很適合第一天的工程觀念：系統應該用規格降低錯誤，而不是期待每個人永遠填對。

## 維護原則

改版時請優先維持：

1. Prep Course 範圍清楚，不滑向正式專案。
2. 學員不用手寫複雜 JSON。
3. profile 欄位少，公開風險低。
4. 前端可讀、可修、可被小範圍修改。
5. Agent 輸出可以被 validate 和 diff 檢查。
6. 文件按讀者分層，不記錄私人對談脈絡。
